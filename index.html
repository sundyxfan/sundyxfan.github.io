git reset && revert 回退到某个版本

场景：

  • 改完代码提交，上线发现有问题，回滚
  • 改完代码测试完成，上线后发现你的修改影响可之前的代码

这是开发常见的问题，所以git的取消提交
大致分为下面两种情况：

  • 没有push

这种情况发生在你的本地代码仓库，可能你add, commit以后发现有问题，准备取消提交，用到下面命令：

–mixed 默认模式，会保留源码，只是将git commit和index信息退回到某个版本
–soft 保留源码，值回退到commit对应版本，不涉及index的回退，如果需要提交，直接commit即可
–hard 源码也会回到某个版本，commit index都回到某个版本, 当然有人在push代码以后，也使用git reset --hard XXX 回退代码到某个版本，但这样有个问题是，你献上的代码没有变，线上commit index都没有变，当你把本地代码修改完提交的时候你会发现全是冲突，所以这时就的使用下面的方式

  • 已经push
    对于已经把代码push到线上仓库，你回退本地代码其实也想同时回退线上的代码，回滚到某个指定的版本，线上，线下代码保持一致，你要用到下面的命令：

commitID回滚版本的唯一commit标示，
revert之后的本地代码会回滚到指定的历史版本，这时再git push即可把线上代码更新，不会像reset造成冲突问题

上面两种方案看似一样，但是完全不一样

  • 上面我们说的如果你已经push到线上代码库，reset删除指定commit以后，你git push 可能导致一大堆冲突，但git revert并不会
  • reset 是在正常的commit历史中,删除了指定的commit,这时 HEAD 是向后移动了,而 revert 是在正常的commit历史中再commit一次,只不过是反向提交,他的 HEAD 是一直向前的.

require 用法系列-

平时在做基于node开发时，遇到require(‘xx’),require(‘./XX’)

  • xx是内置模块

    比如require(‘http’),直接返回该模块

  • xx 以”./“ or “/“开头

    根据x所在的模块确定x的路径，依次查找下面文件，只要有一个存在，就返回，不在继续执行

  • xx是目录，依次查找目录，只要其中有一个存在，就返回该文件

  • xx不带路径

    根据x所在的父模块，确定x可能的安装目录

  • 抛出“not found”

node-debug

nodejs 调试原理

几乎所有的IDE自带的调试功能都是利用nodejs的Debugger内建模块，在这基础上进行封装。

node –debug-brk=5858 xx.js 告诉nodejs在打开的时候启动了内建调试功能，并且监听制定的端口5858过来调试命令。
Debugger listening on port 5858
可以访问localhost:5858

node-inspector –web-port 8080 –debug-port 5858
–web-port 是chrome devtools的调试页面地址端口，–debug-port为nodejs启动的内建debug端口，我们访问localhost:8080/debug?port=5858开打开页面。更多设置可以参考node-inspector官方文档。

前端基础篇

前端是一种技术问题较少、工程问题较多的软件开发领域。

大体量：多功能、多页面、多状态、多系统；
大规模：多人甚至多团队合作开发；
高性能：CDN部署、缓存控制、文件指纹、缓存复用、请求合并、按需加载、同步/异步加载、移动端首屏CSS内嵌、HTTP 2.0服务端资源推送。
