git reset && revert 回退到某个版本

场景：

  • 改完代码提交，上线发现有问题，回滚
  • 改完代码测试完成，上线后发现你的修改影响可之前的代码

这是开发常见的问题，所以git的取消提交
大致分为下面两种情况：

  • 没有push

这种情况发生在你的本地代码仓库，可能你add, commit以后发现有问题，准备取消提交，用到下面命令：

–mixed 默认模式，会保留源码，只是将git commit和index信息退回到某个版本
–soft 保留源码，值回退到commit对应版本，不涉及index的回退，如果需要提交，直接commit即可
–hard 源码也会回到某个版本，commit index都回到某个版本, 当然有人在push代码以后，也使用git reset --hard XXX 回退代码到某个版本，但这样有个问题是，你献上的代码没有变，线上commit index都没有变，当你把本地代码修改完提交的时候你会发现全是冲突，所以这时就的使用下面的方式

  • 已经push
    对于已经把代码push到线上仓库，你回退本地代码其实也想同时回退线上的代码，回滚到某个指定的版本，线上，线下代码保持一致，你要用到下面的命令：

commitID回滚版本的唯一commit标示，
revert之后的本地代码会回滚到指定的历史版本，这时再git push即可把线上代码更新，不会像reset造成冲突问题

上面两种方案看似一样，但是完全不一样

  • 上面我们说的如果你已经push到线上代码库，reset删除指定commit以后，你git push 可能导致一大堆冲突，但git revert并不会
  • reset 是在正常的commit历史中,删除了指定的commit,这时 HEAD 是向后移动了,而 revert 是在正常的commit历史中再commit一次,只不过是反向提交,他的 HEAD 是一直向前的.
